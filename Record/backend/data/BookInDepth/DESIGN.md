# Book In Depth — 设计说明书

> 给后续协作的 AI / 写作者读：在为本 app 写「精读稿」之前请先读完这份文件。它定义了这个 app **不是什么**、**应该是什么**、以及一份稿子的**质量基线**。

---

## 1. 这个 app 是怎么来的

它是 [`A Book a Day`](../ABookADay/) 的兄弟工具，是为了解决 A Book a Day 的一个具体问题而诞生的：

> A Book a Day 把一本书压缩到大约 **2000 字**，太多内容被忽略掉了。
> 读起来「太简单了、太简化了」。
>
> Book In Depth 反其道——**一本书 ~10000 字**，把书真正讲透。

两个 app 共存，不是冗余：

| | A Book a Day | Book In Depth |
|---|---|---|
| 定位 | 5 分钟搞懂一本书 | 一小时把书读到底 |
| 字数目标 | ~2000 字 / 本 | **~10000 字 / 本** |
| Tab 数 | 7（一句话/概要/要点/思维导图/金句/笔记/朗读稿） | **2（思维导图 + 朗读稿）** |
| 右栏 | 资料文件夹 + AI 模板 | **无** |
| TTS | 有（Edge TTS 整本朗读） | **无（用来看，不是用来听）** |
| 主用例 | 快速扫一遍、随手听 | 静下来精读、消化 |

**两个 app 数据完全独立**——
A Book a Day 的数据在 `backend/data/ABookADay/`，
Book In Depth 的数据在 `backend/data/BookInDepth/`（本目录）。
书可以同时存在于两边，也可以只存在于其中一边。一般来说：

- 一本「我快速过一遍就够」的书 → 只在 A Book a Day 里。
- 一本「值得读到骨头里」的经典 → 在 Book In Depth 里专门写一篇万字稿。

## 2. 一本书的目录结构

```
backend/data/BookInDepth/
├── shelves.json                       # 书架定义（想读/在读/已读/收藏/...）
├── DESIGN.md                          # 本文件
└── <book-id>/                         # 每本书一个文件夹
    └── book.json                      # 元信息 + 两个 markdown section
```

**没有 `materials/` 子目录**——Book In Depth 不接受附件、不接受音频。
**没有 `_audio/` 目录**——没有 TTS。
书的唯一存在形式是 `book.json`。

### `book.json` 的结构

```json
{
  "id": "sapiens-harari",
  "shelfId": "reading",
  "title": "人类简史",
  "author": "尤瓦尔·赫拉利",
  "originalTitle": "Sapiens: A Brief History of Humankind",
  "year": "2011",
  "language": "zh",
  "pages": "464",
  "isbn": "...",
  "tags": ["历史", "人类学", "经典"],
  "rating": 0,
  "startedAt": "",
  "finishedAt": "",
  "myTake": "一句话的个人评价（最多 500 字符）",
  "createdAt": "2026-06-02T...",
  "updatedAt": "2026-06-02T...",
  "sections": {
    "mindmap":   "...（缩进式 Markdown）...",
    "narration": "...（深度精读稿 ~10000 字 Markdown）..."
  }
}
```

后端只接受 `sections` 里 `mindmap` 和 `narration` 这两个 key——
任何其它 key 都会被静默丢弃，所以**不要**写 `oneLiner` / `synopsis` / `points` / `quotes` / `notes` 之类的字段。

## 3. 「思维导图」section 的写法

- 缩进式 Markdown 列表。**根节点是书名**，下一层是「整本书的骨架」。
- 用 `- ` 前缀，缩进 **2 空格** 表示子节点。
- 前端会自动解析成可视的树（彩色节点 + 虚线连线）。
- **目标深度 ≥ 3 层，不超过 5 层。** 一级 = 全书结构，二级 = 章 / 主题，三级 = 关键概念，四级 = 例子或子论点。
- 节点文字简短、概念化——这里是骨架，不是叙事。叙事留给朗读稿。
- 写完后回头检查：思维导图本身就应该可以让一个完全没读过这本书的人**5 分钟内**抓住全书结构。

### 示例

```markdown
- 人类简史
  - 三大革命
    - 认知革命（7 万年前）
      - 想象虚构的能力
      - 围绕共同故事大规模合作
    - 农业革命（1.2 万年前）
      - 「人类最大的骗局」
      - 小麦驯化了我们
    - 科学革命（500 年前）
      - 承认「我们不知道」
      - 科学 + 帝国 + 资本主义
  - 三大统一力量
    - 金钱（最普世的虚构）
    - 帝国（最稳定的政治）
    - 宗教（最深的合作粘合剂）
```

## 4. 「朗读稿」section 的写法 ★ 最关键

> 这是本 app 的核心交付物。质量基线见下面。

### 4.1 字数 & 形态

- **目标：~10000 个中文字符**（用户的明确要求）。允许 ±20% 浮动，下不低于 8000，上不高于 12000。
- **是 Markdown，不是纯文本。**
- 用 `## 一级标题` 划分章节——**前端会自动从 `##` 抽取目录**，渲染在朗读稿左侧作为可点击的导航。所以**好的章节标题 = 好的目录**。
- 不需要在最前面加 `# 书名`——书名已经在卡片元信息里了。可以从一个 `# 引言` / `# 序章` 开始，也可以直接进入第一个 `## 章节`。

### 4.2 命名："朗读稿" 是历史名字，**实际上是深度精读稿**

用户说："我是用来看的，不是用来听的。"

不要写成口语化的散文（那是 A Book a Day 的 `narration` 字段的做法）。
**这里要写成可以被反复阅读的精读散文**——可以分段、可以列点、可以引用、可以加小标题。
风格上偏向：
- **梁文道 / 罗振宇风格的"讲书"** — 通俗但深入
- **《纽约客》/《大西洋月刊》的长篇书评** — 有立场、有上下文
- **得到 / 看理想的精品课讲义** — 结构清晰、可消化

不是教科书摘要，不是百度百科条目。

### 4.3 一份 10000 字精读稿的推荐结构

不是死板的模板，是骨架。可以根据书的类型调整。

```
# 引言（约 600–1000 字）
- 一句话钩子（这本书到底在回答什么问题）
- 作者是谁、什么背景写下这本书、什么时代背景
- 这本书在它的领域里的位置（前驱、对话对象、影响）
- 为什么今天还值得读

## 全书脉络（约 1000–1500 字）
- 把书的核心论证骨架讲清楚
- 全书要解决的核心问题 + 作者的核心主张
- 论证的几条主线（不展开，只给地图）

## 第一部分：xxx（约 1500–2000 字）
（沿着思维导图的二级节点，每一节展开讲）
- 作者的论点
- 关键概念的定义
- 至少一个生动的例子或类比
- 这个论点和别处（其它书 / 现实 / 历史）的联系

## 第二部分：xxx（约 1500–2000 字）
…

## 第三部分：xxx（约 1500–2000 字）
…

（书的主体部分一般 3–5 个大块，加起来 5000–7000 字）

## 几条对照与延伸（约 800–1500 字）
- 这本书和领域内 2–3 本经典的对比
- 哪些地方是它的独特贡献，哪些地方有局限
- 它对你（读者）今天的生活的实际意义

## 落地：读完可以做的一件事（约 300–600 字）
- 给读者留一个可以立刻动手的小练习
- 不是鸡汤口号，是具体动作
```

### 4.4 写作的质量基线（10 条 must-have）

1. **有立场。** 不是百科条目式的中立报告。读者读完应该能感受到「写稿人对这本书的判断是什么」。
2. **有 3 个以上的具体例子 / 故事 / 类比。** 抽象论点必须落到具体。一段全是抽象概念的文字 = 不及格。
3. **有跨书 / 跨领域的对照。** 一本书不是孤岛——至少要点到 2 本相关的书 / 思想家 / 现象。
4. **有「为什么是这一本」的论证。** 同类书有十几本，为什么这本值得花 10000 字？答案要在稿子里。
5. **思维导图 ⊆ 朗读稿。** 朗读稿必须覆盖思维导图里的每一个二级节点。反之不必——朗读稿可以加思维导图里没有的延伸。
6. **段落短。** 中文长段读起来很累。一段不超过 5 行（视觉行）。
7. **不要堆引文。** 引文可以有，但每章不超过 2 段，且要解释为什么这句重要。
8. **不要"内容农场感"的衔接句**——"那么接下来我们来看…"、"以上就是…的全部内容了"。这是精读稿，不是 PPT 解说。
9. **有时间感。** 这本书是哪一年的？放在它的时代里读出什么？放在今天读又出什么？
10. **结尾要落地。** 不能只总结。给读者一个具体动作 / 一个新问题 / 一个值得记住的句子。

### 4.5 Markdown 语法可用清单

前端的渲染器是轻量的（手写解析，不是 react-markdown）。**可用**：

- `# / ## / ### / ####` 标题（生成 TOC 锚点）
- `- / * / 1.` 列表
- `>` 引用块
- `**粗体** / *斜体* / `行内代码` / [链接](url)`
- 段落（连续行 = 一段）

**不可用 / 慎用**：
- 表格（解析器不支持 — 别用 `|---|` 语法）
- HTML 标签（不解析）
- 图片（没有图床）
- 多级嵌套列表（会渲染但视觉上不优雅，长稿子里别用太多）

## 5. 后续往这个 app 里加书的流程

1. **选书。** 用户会指定 1–N 本书。优先选「值得反复读的经典 / 厚书 / 影响过自己的书」。
2. **创建文件夹。** 在 `backend/data/BookInDepth/` 下新建 `<book-id>/` —— id 是书名的英文 / 拼音 slug，参考 `ABookADay/` 里现有命名（如 `sapiens-harari`、`mindset-dweck`）。
3. **写 `book.json`。** 元信息齐全 + 两个 sections（mindmap + narration）。可以参考下面的「最小骨架」。
4. **不要在文件夹里放任何其它文件。** 没有 materials/、没有图片、没有 PDF。这个 app 就只接受 `book.json`。
5. **写完后做一遍自检：**
   - 朗读稿字数（中文字符）在 8000–12000 之间吗？前端会在元信息里显示字数。
   - 思维导图所有二级节点都在朗读稿里展开了吗？
   - 朗读稿至少有 4 个 `##` 章节标题吗？（这样左栏 TOC 才有用）
   - 上面 4.4 的 10 条质量基线都过了吗？

### `book.json` 最小骨架

```json
{
  "shelfId": "wantToRead",
  "title": "（书名）",
  "author": "（作者）",
  "originalTitle": "",
  "year": "",
  "language": "zh",
  "tags": [],
  "myTake": "",
  "sections": {
    "mindmap": "- （书名）\n  - 一级分支\n    - 二级分支\n",
    "narration": "# 引言\n\n…\n\n## 第一章\n\n…"
  }
}
```

`id` / `createdAt` / `updatedAt` 不用填——首次通过 API 创建时后端会补；如果是直接放文件，下次 app 启动时会自动 normalise。

## 6. 技术参考（前端 / 后端 行为）

- 后端：`backend/apps/book_in_depth/routes.py`，挂载在 `/api/book-in-depth`。
- 前端：`frontend/src/apps/book_in_depth/App.jsx` + `styles.css`。
- API 形态和 A Book a Day 几乎一致：`/library`、`/shelves`、`/books`、`/books/<id>`。
- **没有 `/tts` 路由**，**没有 `/materials` 路由**。
- 前端左栏每本书会显示一个 `Xk` 小标签，代表朗读稿的字数（向下取整千），方便一眼看出"这本是不是真写到了万字"。

## 7. 一句话哲学

> A Book a Day 是「**广**」——多读、快读、宽口径。
>
> Book In Depth 是「**深**」——少读、慢读、读到底。
>
> 两个 app 的本质区别不是 UI，是**对一本书该投入多少**的不同主张。
> 给本 app 写稿时，你的对照标准不是「写得清楚」——是「写得让一个看过原书的人也愿意再读一遍」。
