# Book In Depth — 设计说明书

> 这是「写什么」的文件。「怎么写」看 METHODOLOGY.md，「自动化流程」看 WORKFLOW.md。
> 三个文件互相补充，三者都不可绕过。

---

## 1. 这个 app 的定位

把一本经典书——**客观、准确、详细地复述出来**。

不是「2000 字浓缩 + 个人感悟」。
不是「读书笔记 + 行动指南」。
不是「博主式讲书」。

而是——

**一本书的客观提炼稿，每本 ~10000+ 字，包含原书的核心论点、论证链条、关键概念、具体例子、作者立场——读者读完应该说「这就是这本书在讲什么」，而不是「这是作者对那本书的看法」。**

---

## 2. 一本书的目录结构

```
backend/data/BookInDepth/
├── DESIGN.md                          # 本文件（写什么）
├── METHODOLOGY.md                     # 怎么写（角色、流程）
├── WORKFLOW.md                        # 自动化流程
├── LOOP_PROMPT.md                     # 循环触发时的 prompt
├── books_queue.xlsx                   # 用户编辑的入口
├── queue_helpers.py                   # 队列工具
├── shelves.json                       # 书架定义
└── <book-id>/                         # 每本书一个文件夹
    └── book.json                      # 元信息 + 两个 markdown section
```

文件夹**只放** `book.json` —— 不接受 PDF、不接受音频、不接受附件。

### `book.json` 的结构

```json
{
  "id": "sapiens-harari",
  "shelfId": "reading",
  "title": "人类简史",
  "author": "尤瓦尔·赫拉利",
  "originalTitle": "Sapiens: A Brief History of Humankind",
  "year": "2011",
  "language": "zh",
  "pages": "464",
  "isbn": "...",
  "tags": ["历史", "人类学", "经典"],
  "rating": 0,
  "startedAt": "",
  "finishedAt": "",
  "myTake": "一句话总结（最多 500 字符——可以是「这本书的核心命题是 X」）",
  "createdAt": "...",
  "updatedAt": "...",
  "sections": {
    "mindmap":   "...（缩进式 Markdown）...",
    "narration": "...（提炼稿 ~10000+ 字 Markdown）..."
  }
}
```

后端只接受 `sections` 里 `mindmap` 和 `narration` 这两个 key——任何其它 key（`synopsis` / `points` / `quotes` / `notes` 之类）都会被静默丢弃。

## 3. 「思维导图」section 的写法

- 缩进式 Markdown 列表。**根节点 = 书名**。下一层是「整本书的骨架」。
- `- ` 前缀，缩进 **2 空格** 表示子节点。
- 前端自动解析成树状视图。
- **深度 3–5 层**。一级 = 全书结构，二级 = 章 / 主题，三级 = 关键概念，四级 = 例子 / 子论点。
- **节点文字 = 概念**，不是叙述。叙述放朗读稿。
- 写完后回头检查：**没读过这本书的人看思维导图，5 分钟能不能抓住全书结构**？

---

## 4. 「朗读稿」section 的写法 ★ 最关键

> 「朗读稿」是历史名字。实际上是**深度提炼稿**——用户的原话「我是用来看的，不是用来听的」。

### 4.1 字数 & 形态

- **质量基线**：narration 字段的 CJK 字符 ≥ **10,000**。
- 是 Markdown，不是纯文本。
- 用 `## 一级标题` 划分章节——前端从 `##` 抽取目录。所以**好的章节标题 = 好的目录**。
- 不需要在最前面 `# 书名`——元信息已经在卡片里。可以直接 `# 引言` 开始。

### 4.2 内容比例（METHODOLOGY.md 第 0 条铁律的应用）

- **~80%** 是**原书内容的客观复述**——论点、论据、推理链、概念定义、具体例子、作者引文
- **~15%** 是**必要背景**——作者生平、写作动机、时代位置、关键术语解释（**只为理解原书内容服务**）
- **~5%** 是**诚实的局限说明**——哪些被批评、哪些过时、哪些不严密（**也是客观陈述**，不是个人评价）

### 4.3 一份提炼稿的推荐结构

不是死板模板，是骨架。按书的形态调整。

```
# 引言（约 600–1000 字）
- 这本书在论证什么（一句话）
- 作者是谁，在什么背景下写
- 这本书在它的思想传统 / 行业里的位置
- 全书的核心命题（不展开，先给地图）

## 作者与写作背景（约 800–1500 字）
- 作者生平的关键节点（与写这本书相关的部分）
- 写作动机
- 关键社会 / 学术语境

## 全书结构（约 600–1000 字）
- 全书几部分 / 几章
- 每部分讲什么（一句话）
- 论证脉络如何展开

## 核心概念 / 论点深度展开（5000–8000 字，占大头）
- 沿着思维导图的二级节点，一节一节展开
- 每节内部：
  - 作者的论点（精确陈述）
  - 关键概念的定义（用作者自己的话）
  - 作者用的证据 / 例子 / 案例
  - 作者怎么从前提推出结论
  - 作者怎么回应可能的反驳
- ★ 这是稿子的主体——「这本书在讲什么」全在这里

## 接受史 / 影响（约 600–1200 字）
- 这本书出版后的反响（谁推崇 / 谁批评）
- 后续学者 / 实践者怎么继承 / 修正 / 反驳
- 对相关领域的实际影响（不是「读者收获」，是历史影响）

## 局限 / 批评（约 400–800 字）
- 学界对这本书最尖锐的批评是什么
- 时代局限（哪些数据 / 案例已过时）
- 作者自己后来承认的错误（如果有）

## 一句话总结（约 100–200 字）
- 「这本书的核心命题是 X」式的结尾
- 不是「愿你...」，是把整本书的论证浓缩成最短的可记忆形式
```

### 4.4 写作的质量基线（10 条 must-have）

1. **客观复述。** 读者读完应该能复述书的内容——而不是记住你的感受。
2. **论证链条完整。** 作者从前提到结论的每一步——前提 → 中间推论 → 结论——都讲清楚。**不能跳步**。
3. **概念有定义。** 每个核心概念出现时，用作者自己的话给出定义——而不是你的概括。
4. **例子是作者用的例子。** 优先复述书中的例子，不要替作者编例子。如果加了对当代读者更熟悉的例子，**明确标注「这是为帮助理解加的，不是书里的」**。
5. **跨书联想要克制。** 只在作者本人引用了别的书时才提那本书。**不要替作者建立他没建立的联系**。
6. **段落短。** 中文读者对长段疲劳。每段不超过 5 行视觉行。
7. **不要堆引文。** 引文可以有，但每章不超过 2 段，且必须**说明为什么这句重要**——是核心命题？是关键例证？还是反复出现的口号？
8. **不要"内容农场感"衔接句**——「那么接下来我们来看…」、「以上就是…」全删。直接进下一个论点。
9. **有时间感。** 这本书写于哪一年，作者多大，那个年代的核心议题是什么——这些为了**让读者理解原书的语境**而提供。
10. **结尾是「这本书的命题」，不是「愿你...」。** 不要励志总结。最后一段要把整本书的论证收束成可记忆的形式。

### 4.5 严禁的写法（METHODOLOGY 第 0 条铁律延伸）

- ❌ 「读完这本书希望你...」、「愿你...」、「让我们一起...」式抒情结尾
- ❌ 「我的判断是...」、「我认为...」式个人立场（除非明确标注为局限说明的一部分）
- ❌ 「这本书让我想到了 X 那本书」式跨书联想（除非作者本人引用了 X）
- ❌ 「行动指南」「实践清单」「7 步法」（除非书本身就是这种实用书）
- ❌ 编造作者没说的话、替作者加观点
- ❌ 「读完这本书你应该...」式给读者下指令
- ❌ 把书的内容硬塞进当代政治 / 文化语境去解读（除非原书本身就是当代书）

### 4.6 允许 / 鼓励的写法

- ✅ 大量复述作者的论点、定义、例子
- ✅ 详细展开作者的论证链条
- ✅ 引用作者的原话（注明是引用）
- ✅ 说明作者的写作背景以帮助理解（不是为了「拉关系」）
- ✅ 客观陈述接受史和后世影响（用事实，不用判断）
- ✅ 客观陈述学界批评（用别人的批评，不用自己的）

### 4.7 Markdown 语法可用清单

前端的渲染器是轻量的（手写解析，不是 react-markdown）。**可用**：

- `# / ## / ### / ####` 标题（生成 TOC 锚点）
- `- / * / 1.` 列表
- `>` 引用块（**用于引用原文**）
- `**粗体** / *斜体* / ` 行内代码 `/ [链接](url)`
- 段落（连续行 = 一段）

**不可用 / 慎用**：
- 表格（解析器不支持 — 别用 `|---|` 语法）
- HTML 标签（不解析）
- 图片（没有图床）
- 多级嵌套列表（会渲染但视觉上不优雅）

## 5. 元信息字段（卡片显示）

| 字段 | 必填 | 用途 |
|---|---|---|
| `title` | ✅ | 中文标题 |
| `author` | ✅ | 作者名（多个用 ` / ` 分隔） |
| `originalTitle` | 推荐 | 原文标题（西文书） |
| `year` | 推荐 | 出版年（或多个版本，如「1543（拉丁原版）」） |
| `language` | ✅ | `zh` |
| `pages` | 可选 | 中文版页数 |
| `isbn` | 可选 | 推荐版本的 ISBN |
| `tags` | 推荐 | 3–6 个，便于检索（如 `["哲学", "现实主义", "经典"]`） |
| `myTake` | 可选 | **一句话客观陈述这本书的核心命题**，≤500 字符。不是个人感受，是浓缩的命题。 |

`shelfId` 默认 `wantToRead`，处理完成后由用户/Claude 调整。

## 6. 后续加书的流程（人工流程，自动流程见 WORKFLOW.md）

如果你是 Claude 手动操作（不通过队列）：

1. 在 `backend/data/BookInDepth/` 下新建 `<book-id>/` —— id 是书名的英文 / 拼音 slug（参考现有命名，如 `sapiens-harari`、`mindset-dweck`）。
2. 在文件夹里只放 `book.json`。
3. 元信息齐全 + 两个 sections（mindmap + narration）。
4. 写完后做一遍自检：
   - narration 的 CJK 字符 ≥ 10,000 吗？
   - 思维导图所有二级节点都在朗读稿里展开了吗？
   - 朗读稿至少有 5 个 `##` 章节标题吗？
   - 上面 4.4 的 10 条质量基线都过了吗？
   - 4.5 的严禁清单一条都没犯吗？

不达标 → 删掉重写（参见 METHODOLOGY 反「补丁式写作」铁律）。

## 7. 技术参考

- 后端：`backend/apps/book_in_depth/routes.py`，挂载在 `/api/book-in-depth`。
- 前端：`frontend/src/apps/book_in_depth/App.jsx` + `styles.css`。
- 路由：`/library`（书架 + 书列表）、`/queue`（队列）、`/shelves`、`/books`、`/books/<id>`、`/trigger`（立即触发）。
- 队列文件：`books_queue.xlsx`（用户编辑入口）。
- 前端左栏每本书会显示 `Xk` 标签（朗读稿字数向下取整千），一眼看出「这本是不是真写到了万字」。

## 8. 一句话定位

> **客观、详细、结构完整地复述一本书——读者读完知道「这本书在讲什么」，而不是「写稿人怎么想这本书」。**
>
> 不是博主讲书，不是个人读书笔记，是**学术级的客观提炼**——以普通受教育者能读懂的语言。
